--- Root Cause Analysis ---

  The CachedLLMClient computes cache keys via SHA-256 of
  (model + systemPrompt + userPrompt). The cache key is:
    hash.update(request.model)
    hash.update(request.systemPrompt)
    hash.update(request.userPrompt)

  The userPrompt includes "Segment ID: <segmentId>" where segmentId
  is generated by randomUUID() in src/parser/ui-segmenter.ts (lines
  494, 519, 545). Since segment IDs are re-generated on every run,
  the SHA-256 hash changes, causing 100% cache misses on Run 2.

  Evidence:
  - Run 1: 90 cache entries written
  - Run 2: 90 NEW cache entries written (total 180 after both runs)
  - Zero overlap between Run 1 and Run 2 cache keys

--- Required Fix (Wave 3 scope) ---

  The cache key computation must EXCLUDE non-deterministic prompt
  components. Two approaches:

  Option A (Recommended): Strip the Segment ID from the prompt before
  computing the cache key. The Segment ID has no semantic value for
  the LLM; it is only used for traceability.

  Option B: Use a deterministic segment ID (hash of DOM content)
  in ui-segmenter.ts instead of randomUUID().

  IMPORTANT: The CachedLLMClient implementation itself is correct.
  The issue is in the prompt construction, not the cache layer.
